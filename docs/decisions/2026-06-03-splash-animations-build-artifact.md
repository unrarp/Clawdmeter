# splash_animations.h is a build artifact, not a committed file

## Context
`firmware/src/splash_animations.h` is an ~11 MB generated C header (palette +
per-frame cells) from the SVG→capture→binary→header pipeline. The project used to
commit both the header and its JSON source (~26 MB per migration), had to manually
exclude it from review (binary-as-C-array is unreadable), and regenerate it after
any animation change.

## Decision
`splash_animations.h` is gitignored and regenerated by a PlatformIO `pre:`
extra-script hook (`firmware/scripts/gen_splash.py`, registered in all three
`[env:...]` blocks) before every `pio run`. The committed source of truth is
`tools/svg_anim_data/manifest.json` + `clawd_*.bin` (raw uint8 palette indices,
~3.8 MB total). The stdlib-only Python generator needs only PlatformIO's bundled
Python — no node, no extra installs on a fresh checkout.

## Why
- **Repo size.** Header (~11 MB) + JSON (~15 MB) = ~26 MB per migration vs ~3.8 MB
  binary + gitignored header.
- **Binary data isn't git-diffable.** Frame data as C text arrays has no meaningful
  diff; `.bin` files at least keep the repo honest about what's diffable.
- **It's the firmware's `dist/`.** You don't commit compiled output; PlatformIO
  `extra_scripts` is the pre-build equivalent of `npm run build`.

## Alternatives considered
- **Commit header, gitignore JSON** — still grows the repo ~11 MB per migration.
- **Commit as JSON (the previous state)** — ~15 MB verbose; the binary is identical
  data at 3.8 MB.
- **Keep the node `convert_to_c.js` converter** — required node on every build; the
  Python generator removes that from the flash path.

## Prevention
- It's in `.gitignore`; `git rm --cached firmware/src/splash_animations.h` was needed
  to untrack it (gitignoring a tracked file alone has no effect).
- The hook logs `regenerating` / `up to date` on every build so the state is visible.
